# Contributing to master-instructional-design

Contributions are welcome. This skill is a living document — the L&D field evolves, frameworks get updated, and new research emerges. If you find something that should be added, corrected, or improved, please open an issue or pull request.

---

## What makes a good contribution

**Good candidates:**
- A framework, model, or research finding that belongs in the canon but isn't here
- An outdated reference that has a newer edition or has been superseded
- A factual error in how a theory or model is described
- A missing engagement mode for a domain the skill should cover
- A cross-reference gap between reference files
- A gap identified through real use (bring the test case)

**Not a good fit:**
- Opinion-based preference changes ("I prefer X framework over Y")
- Additions that duplicate content already covered
- Content that reproduces copyrighted text verbatim
- Promotional content for commercial tools or services

---

## How to contribute

### Reporting an issue
Open a GitHub Issue with:
- Which file the issue is in (SKILL.md or which reference file)
- What the problem is (factual error, missing content, outdated reference, etc.)
- If you have a suggested fix, include it

### Submitting a pull request
1. Fork the repository
2. Make your changes in a new branch: `git checkout -b fix/description-of-change`
3. Keep changes focused — one issue per PR
4. Update cross-references in other files if your change creates new connections
5. Run the skill packager to confirm it still validates: the SKILL.md description must stay under 1024 characters
6. Open a pull request with a clear description of what changed and why

---

## IP and attribution standards

This skill describes frameworks, models, and published works in original language. All contributions must:
- Paraphrase rather than reproduce copyrighted text
- Attribute theories and models to their originators by name and date
- Use tool and product names only for factual reference — no endorsement language
- Not reproduce proprietary methodologies in ways that substitute for the original work

---

## Style conventions

- All content is in original prose — no copied text from external sources
- Framework names are used factually (e.g., "Gagné's nine events of instruction") not as branded properties
- ID application notes use the format: **ID application**: [practical implication]
- Cross-references use the format: `references/filename.md §section-number`
- Dimension labels in the audit framework use letters A–I
